---
title: Corpus Import Run
description: Workflow for batch imports and report review.
---

Use this workflow when running StackImport against a collection of stacks,
checking parser coverage across many inputs, or generating evidence for a
format-investigation task.

## Prerequisites

- Build `stackimport` first.
- Confirm the input directory. The default local corpus is
  `/Users/jrepp/d/pantechnicon/stacks`.
- Ensure `uv` is available for the script shebang.
- Ensure archive tooling such as `unar` is available when processing `.sit` or
  `.hqx` inputs.

## Steps

1. Run the importer script from the repository root:

   ```sh
   scripts/import_all_stacks.py --stackimport-bin build/stackimport
   ```

   Captured logs, extracted archives, packages, reports, and the run database
   default to `/tmp/stackimport-import-runs/<run-id>/`. Use `--run-root` only
   when a persistent output location is intentional.

   The scanner skips known non-input directories by default, including `.git`,
   `import-runs`, `roms`, and `node_modules`. Add more directory basenames with
   repeated `--ignore-dir <name>` options, or use `--no-default-ignore-dirs`
   when investigating the scanner itself.

2. For focused analysis, prefer a small input subset before running the full
   corpus.
3. Review the generated run directory under `/tmp/stackimport-import-runs/`.
4. Inspect report tables in the run SQLite database, captured logs, output
   packages, and aggregate stats.
5. Identify whether failures come from extraction, file classification, parser
   behavior, media conversion, or output indexing.
6. Use `--dumprawblocks` on representative stacks before changing parser logic.

## Embedded Conversion Follow-Up

After a completed run, convert embedded PBM/P4 files when needed:

```sh
scripts/convert_embedded_files.py --run-db import-runs/<run-id>/run.db
```

Preserve generated `*.provenance.json` sidecars. They record source hashes,
target hashes, conversion status, and stack/archive metadata.

## Validation

For corpus-driven parser changes, re-run the same focused subset after the fix.
Use the full C++ validation workflow before handing off code changes.

## Handoff

Report the run directory, input scope, command used, key database/report tables
reviewed, failure categories found, and any representative stacks selected for
follow-up.
