Resource conversion is driven by `StackImportResourceTransforms.cpp` and exposed
through generated package artifacts plus optional C API resource payload
callbacks. Native resource metadata is recorded in `source-manifest.json`.

Unsupported resource types are still preserved in native resource metadata and
raw package artifacts when resource-fork extraction sees them. New conversions
should be added through the transform layer so package output and C API payload
callbacks stay aligned.

## ROM Resource Mapping Notes

ROM mode reuses this conversion layer after it maps ROM-resident resources. The
current mapper recognizes three resource sources:

- Standard embedded resource forks, when a full resource-fork header/map is
  present.
- Older inline ROM resources, using the 16-byte link/payload prefix plus
  adjacent type, ID, flags, optional Pascal name, payload offset, and payload
  length fields.
- ROM-packed `Kurt` resources, using nearby type, ID, flags, optional Pascal
  name, alternating `kc`/`ck` padding, wrapper lengths, and payload bytes.

`analysis.json.resources` and `resources.tsv` record `length`,
`stored_length`, `expected_length`, and `wrapper_format` so packed/wrapped ROM
records remain distinguishable from ordinary resource-fork records.

ROM mode can also write a self-contained resource index with
`--emit-resource-index`. The index is emitted as `resource-index/index.json`
with raw and converted files under `resource-index/resources/`. Artifact paths
inside that index are relative to the `resource-index/` directory so the folder
can be reused by later conversion tools without requiring the full ROM analysis
tree.
